3 # Animation in Radium
4 
5 The Radium Engine provides a set of classes dedicated to animation,
6 which can be found in `Core/Animation`.
7 
8 In Radium, the animation of any data can be done through the `Ra::Core::Animation::KeyFramedValue`
9 template. It stores the data keyframes, i.e. time-value couples, which are
10 interpolated when the value for a given time is requested.
11 
12 ## Using KeyFramedValue to animate data
13 
14 There are two ways data can be animated using `Ra::Core::Animation::KeyFramedValue`:
15  - When the data is directly specified as a `Ra::Core::Animation::KeyFramedValue`.
16  In this case, particular values at specific times can be specified to create the animation
17  keyframes, and then used when requested:
18 ```c++
19  #include <Core/Animation/KeyFramedValue>
20  #include <Core/Animation/KeyFramedValueInterpolators>
21  struct MyStruct {
22  MyStruct()
23  : m_nonAnimatedData( 2_ra ) // initialize the non animated data as 2
24  , m_animatedData( 0_ra, 1_ra ) { // creating the animated data with value 0 at time 1
25  m_animatedData.insertKeyFrame( 1_ra, 2_ra ); // adding a keyframe with value 1 at time 2
26  }
27 
28  Scalar fetch( Scalar time ) {
29  // fetch the interpolated value for the given time
30  Scalar v_t = m_animatedData.at( t, Ra::Core::Animation::linearInterpolate<Scalar> );
31  // use it
32  return m_nonAnimatedData * v_t;
33  }
34 
35  Scalar m_nonAnimatedData;
36  Ra::Core::Animation::KeyFramedValue<Scalar> m_animatedData;
37  };
38 
39  int main() {
40  MyStruct a;
41  std::cout << a.fetch( 0_ra ) << std::endl; // prints: 0
42  std::cout << a.fetch( 1.5_ra ) << std::endl; // prints: 1
43  std::cout << a.fetch( 3_ra ) << std::endl; // prints: 4
44  }
45 ```
46  - When the data is not specified as a `Ra::Core::Animation::KeyFramedValue` but one wants to animate it.
47  In this case, one must use the `Ra::Core::Animation::KeyFramedValueController` to animate the data:
48 ```c++
49  // keeping the definition of struct MyStruct above:
50  #include <Core/Animation/KeyFramedValueController>
51  struct MyStructAnimator {
52  MyStructAnimator( MyStruct& struct ) {
53  // create the keyframes for the data
54  auto frames = new Ra::Core::Animation::KeyFramedValue<Scalar>( 0_ra, 0_ra );
55  frames->insertKeyFrame( 4_ra, 4_ra );
56  // create the controller
57  m_controller.m_value = frames;
58  m_controller.m_updater = [frames, &struct]( const Scalar& t ) {
59  // fetch the interpolated value for the given time
60  auto v_t = frames->at( t, Ra::Core::Animation::linearInterpolate<Scalar> );
61  // update the data
62  struct.m_nonAnimatedData = v_t;
63  };
64  }
65 
66  void update( Scalar time ) {
67  m_controller.updateKeyFrame( time ); // uses the keyframes to update the data.
68  }
69 
70  Ra::Core::KeyFramedValueController m_controller;
71  };
72 
73 
74  int main() {
75  MyStruct a;
76  MyStructAnimator b( a );
77  std::cout << a.fetch( 0_ra ) << std::endl; // prints: 0
78  std::cout << a.fetch( 1.5_ra ) << std::endl; // prints: 1
79  std::cout << a.fetch( 3_ra ) << std::endl; // prints: 4
80  b.update( 1 ); // now: a.m_nonAnimatedData = 1
81  std::cout << a.fetch( 0_ra ) << std::endl; // prints: 0
82  std::cout << a.fetch( 1.5_ra ) << std::endl; // prints: 0.5
83  std::cout << a.fetch( 3_ra ) << std::endl; // prints: 2
84  b.update( 2 ); // now: a.m_nonAnimatedData = 2
85  std::cout << a.fetch( 0_ra ) << std::endl; // prints: 0
86  std::cout << a.fetch( 1.5_ra ) << std::endl; // prints: 1
87  std::cout << a.fetch( 3_ra ) << std::endl; // prints: 4
88  b.update( 4 ); // now: a.m_nonAnimatedData = 4
89  std::cout << a.fetch( 0_ra ) << std::endl; // prints: 0
90  std::cout << a.fetch( 1.5_ra ) << std::endl; // prints: 2
91  std::cout << a.fetch( 3_ra ) << std::endl; // prints: 8
92  }
93 ```

96 ## Character Animation
97 
98 ### Character Animation Basics
99 
100 In order to animate a digital character, one may choose from two main animation techniques:
101  - skeleton-based animation, in which the character's mesh is bound to an
102 animation skeleton composed of joints, sometimes called bones.
103  - cage-based animation, in which the character's mesh is bound to a set of
104 points defining an englobing deformation cage.
105 
106 In both cases, the character's mesh is bound to the handles (skeleton joints or
107 cage points) through a set of skinning weights.
108 During the animation, the handles transformations are first updated with respect
109 to the animation pose, then the mesh is deformed by combining the handles
110 transformations through the skinnning weights.
111 
112 ### Character Animation in Radium
113 
114 The Radium Engine provides the basic classes for character animation.
115 
116 Starting from a `Ra::Core::Animation::HandleArray`, which represents a deformation metaphor, such as `Ra::Core::Animation::Cage` or `Ra::Core::Animation::Skeleton`,
117 one must associate influence weights to each vertex of a mesh within a `Ra::Core::Animation::WeightMatrix`
118 (rows representing vertices, columns handles).
119 These influence weights, or skinning weights, are then used by a skinning algorithm to deform
120 the associated mesh vertices w.r.t. the handles deformation defined by the animation pose.
121 The mesh is also linked to each handle through a `bindMatrix` that expresses the
122 transformation from the mesh's local frame to the handle local frame.
123 
124 In the case of character animation, an animation is a set of `Ra::Core::Animation::KeyFramedValues<Ra::Core::Transform>`,
125 one for each joint, which are interpolated by default as follows:
126  - the translation and scale channels are linearly interpolated;
127  - the rotation channel is linearly interpolated in quaternion space.
128 
129 Animations are stored in local space, i.e. each handle transformation is expressed
130 in the handle's frame (transformation from bone's parent for skeleton bones,
131 position in cage's frame for cage points).
132 
133 Hence, in order to deform the character's mesh, one must first use the handle's `bindMatrix`
134 to express the mesh vertices position into the handles' local space before combining the
135 handles local transformations to deform the mesh vertices.
136 
137 ### The Skeleton-Based Character Animation Plugin
138 
139 The `SkeletonBasedAnimation` plugin (https://gitlab.com/Storm-IRIT/radium-official-plugins/skeletonbasedanimation)
140 provides one plugin specific to skeleton-based character animation and skinning.
141 This plugin provides a `SkeletonComponent`, which manages user interactions with
142 the animation skeleton, enabling posing the character and playing the animations.
143 It also provides a `SkinningComponent`, which is responsible for deforming the
144 object's mesh vertices according to the desired skinning method.
145 
146 ### Importing skeleton-based character animation data into Radium
147 
148 In order to import animation related data into Radium, the default loader would be the `Ra::IO::AssimpLoader`,
149  which deals with several standard animation formats (fbx, collada, ...).
150 In case one need to develop his own loader because using a file format not managed by the
151 Assimp library, the Radium Engine provides a set of classes in `Core/Asset` defining data
152 wrappers filled by loaders (see the specific documentation -- which does not exist for now).
153 For now, `Ra::Core::Asset::FileLoaderInterface`s in the Radium Engine produce only one `Ra::Core::Asset::FileData` per loaded file.
154 
155 Regarding character animation data, there are a few things to do in order to ensure correct import by the
156 animation and skinning plugins:
157  * The generated `Ra::Core::Asset::FileData` must contain exactly one `Ra::Core::Asset::HandleData` per animated object, as much `Ra::Core::Asset::AnimationData`
158  as needed per animated object and may contain several or no `Ra::Core::Asset::GeometryData`.
159  Note: for objects composed of several mesh parts, these must be referred by their name when filling the `Ra::Core::Asset::HandleData`.
160  * The animation skeleton, i.e. the bones and hierarchy, must be loaded into the
161  `Ra::Core::Asset::HandleData`, as well as the skinning weights.
162  The bone hierarchy will be defined by the `Ra::Core::Asset::EdgeData`.
163  Each bone will be represented by a `Ra::Core::Asset::HandleComponentData`, storing the bone's name,
164  transform in model space and its per-skinned mesh offset matrix and vertex skinning weights.
165  The offset matrix stores the transformation from mesh space to the node's local space.
166  * Animation keyframes, i.e. couples of time and pose (in local space), if any, must be loaded into an `Ra::Core::Asset::AnimationData`.
167  The whole animation must be stored bone-wise in `Ra::Core::Asset::HandleAnimation`s, whose name must match the bone's.
