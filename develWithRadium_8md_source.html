4 There are four main options to work and develop using Radium:
5 1. Write a plugin: full access to the Radium data structures, perfect to implement a new functionality: mesh processing, rendering.
6 2. Write an application: give total control over the GUI, the camera settings, etc...
7 3. Contribute to Radium libraries: to improve/extend core components of Radium.
8 4. Testing: extend radium tests to improve testing coverage or when adding new functionality.
9 
10 Once installed, Radium offers several cmake functions to configure and build your extension.
11 You can access documentation and how-to about these cmake function in the [Radium cmake utilities](@ref cmakeutilities) documentation.

23 # Radium coding style
24 Please follow the scripts/clang-format coding style (tested `with clang-format 9.0`).
25 We also provide a pre commit hook that checks the committed files are correctly formatted.
26 To install both hooks and clang-format, simply run `./scripts/install-scripts-linux.sh` on linux, or adapt to your OS.

28 # Radium test suite
29 ## Preliminaries
30 Radium testing is done using two tools:
31 - CTest: the test system by CMake. It provides compilation, running and basic reporting mechanisms for tests.
32 - Continuous Integration system: Travis.ci and AppVeyor. We use these services to call CTest on several platforms (windows, ubuntu and MacOs) with several compilers, and validate any change made to the main repository.
33 
34 As a Radium contributor, and in most cases, you will have to work only with CTest, and let the CI system run your tests automatically.
35 In this section we describe how to write a new test, how to insert it in the testing suite, and how to validate your results on your computer and the CI systems.
36 
37 ## How to write tests
38 Tests are C++ executables, expected to set their return value accordingly to the test result, e.g. `return EXIT_FAILURE;` or by calling `abort();` in case of failure.
39 
40 Their source code is stored in `Radium-Engine/tests/{Core,Engine,IO,Gui}Tests/src`.
41 
42 We provide in `Radium-Engine/tests/Tests.hpp` some utility functions to ease tests implementation.
43 
44 Let's consider the following snippet:`
45 
46 ~~~{.cpp}
47 #include <Core/Math/LinearAlgebra.hpp> // include path contains Radium sources,
48 #include <OpenMesh/Tools/Subdivider/Uniform/CatmullClarkT.hh> // Radium's dependencies
49 #include <Tests.hpp> // and the testing tools
50 
51 namespace Ra {
52 namespace Testing {
53 
54 void run() {
55  bool condition = true;
56  RA_VERIFY( condition, "This test is passing" );
57  // RA_VERIFY( ! condition, "This test does not pass" );
58 }
59 
60 } // namespace Testing
61 } // namespace Ra
62 
63 
64 int main(int argc, const char **argv) {
65  using namespace Ra;
66 
67  if(!Testing::init_testing(1, argv))
68  {
69  return EXIT_FAILURE;
70  }
71 
72  #pragma omp parallel for
73  for(int i = 0; i < Testing::g_repeat; ++i)
74  {
75  CALL_SUBTEST(( Testing::run() ));
76  }
77 
78  return EXIT_SUCCESS;
79 }
80 ~~~
81 
82 The `Ra::Testing::run()` method is performing the tests, and use the macro `RA_VERIFY` to abort the execution if the condition fails. This macro is provided in `<Tests.hpp>`.
83 
84 The `main()` is simple is should not vary to much across tests.
85 It first initializes the testing helpers (including random number generation), and then call the testing function `g_repeat` times.
86 Repetition is strongly encouraged, especially to test or detect piece of codes involving non-deterministic behavior.
87 Of course, multiple functions can be called in place of the `run()` function shown in this example. A standard use case is:
88 
89 ~~~{.cpp}
90  #pragma omp parallel for
91  for(int i = 0; i < Testing::g_repeat; ++i)
92  {
93  CALL_SUBTEST(( Testing::run<float>() ));
94  CALL_SUBTEST(( Testing::run<double>() ));
95  // ... other types
96  }
97 ~~~
98 
99 Note that we target Unit Tests, which can be defined as:
100 > The purpose of a unit test in software engineering is to verify the behavior of a relatively small piece of software,
101 > independently from other parts. Unit tests are narrow in scope, and allow us to cover all cases, ensuring that every
102 > single part works correctly.
103 >
104 > Source: [https://www.toptal.com/qa/how-to-write-testable-code-and-why-it-matters](https://www.toptal.com/qa/how-to-write-testable-code-and-why-it-matters)

106 ## How to add a test to the testing suite
107 The testing suite is handled by `cmake`: tests need to be added with their sources and dependencies.
108 We provide a `cmake` helper function to add a test easily:
109 
110 ~~~cmake
111  radium_add_test( TARGET my_test_name SRC src/my_test.cpp LIBS radiumCore radiumEngine )
112 ~~~
113 
114 See `Radium-Engine/cmake/ConfigureTesting.cmake` for the implementation of this function, and `Radium-Engine/tests/CoreTests/CMakeLists.txt` for an usage example.
115 
116 Tests are located in `Radium-Engine/tests/{Core,Engine,IO,Gui}Tests/src`. Currently `cmake` scripts are configured only for `Core`, you might need to copy and adapt them in case you add tests for the others packages (this is what you should do ! ;) ).
117
