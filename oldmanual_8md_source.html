4 # Source code organization
5 * `src` contains the main source code of the engine libs
6 
7  * `Core` : the Core module (dynamic library) contains the foundation
8  classes such as math classes, containers, adapters to the standard
9  library, etc.
10 
11  * `Engine` the Engine module (dynamic library) contains all graphics related code and engine subsystems running.
12 
13  * `GuiBase` has the Qt-based GUI classes.
14 
15 * `Plugins` contains the plugins which add Systems and Components to the engine.
16 
17 * `Shaders` contains the OpenGL Shaders used by the renderer.
18 
19 * `Applications` contains applications build with the engine. Currently it contains a minimal example application and the full-fledged `main-app`
20 
21 * `3rdPartyLibs` contains any library dependency. CMake will automatically look in this folder to find the libraries if they are not installed on your system.
22 
23 ## Import and export
24 Core and Engine (and the plugins) are compiled as dynamic libraries, therefore you must be careful about which symbols
25 need exporting, especially on Windows. Every module has a specific header file (`RaCore.hpp` for Core) which must
26 be included first in all other headers of the module. In particular it defines the `*_API` macros which help exporting
27 and importing symbols on Windows.

29 # Core Libs
30 `Core` contains most of the basic code on which the rest of the software is built.
31 * `CoreMacros.hpp` has definitions of basic types, build configuration and useful macros.
32 * `Math` is our math library, which is a wrapper around _Eigen_. The most useful files are `Math.hpp` which has mathematical constants and simple functions, and `LinearAlgebra.hpp` which contains the type definitions of most basic vector and matrix types.
33 * `Containers` has some specially useful std-like containers, most importantly `VectorArray.hpp` which defines a dynamic array of vectors with both a`std::vector`-like interface and compatibility with Eigen.
34 * `Mesh` contains our basic mesh geometry primitives, including the representation of a simple triangle mesh and many functions to operate on geometry.
35 * `String` contains utilities extending `std::string`.
36 * `Log` is a wrapper around the header-only _EasyLogger_ library which allows us to log various events.
37 * `Time` contains utilities around `std::chrono` for precise timings.
38 * `Tasks` contains the definition for the basic Tasks system and the task queue.
39 * `Utils` contains generic utilities such as a Singleton template.

41 # Engine Object Model
42 
43 ## Entities
44 
45 _Entities_ are the base object manipulated by the engine. You can think of them as "game objects", something that
46 represent an object in the scene. Entity data include a world space to object space transform and a list
47 of _Components_
48 
49 The entity's role is only to hold together this transform and the list of components.
50 The entity uses double buffering to prevent transforms from being updated
51 more than once per frame.
52 
53 When creating an entity, if you set its transform, _do not forget_ to call
54 `Entity::swapTransformBuffers`, this might prevent you some headache. Example :
55 ~~~{.cpp}
56 Ra::Engine::Entity* entity = theEntityManager->getOrCreate( "MyEntity" );
57 Ra::Core::Transform transform( Ra::Core::Transform::Identity() );
58 transform.translation = Ra::Core::Vector3( 42, 13, 37 );
59 entity->setTransform( transform );
60 entity->swapTransformBuffers();
61 ~~~
62 
63 ## Systems and Components
64 
65 Each _Component_ represent an aspect of an _Entity_ to a particular engine subsystem.
66 For example, an animated character may have a skeleton(animation component),
67 a high-resolution mesh (display component) and simplified convex shape for collision detection (physics component).
68 
69 Each Component is related to a _System_. The Engine loads several Systems (statically
70 or from a plugin) and they keep the Components of all Entities in the scene updated.
71 
72 For example, an Entity may have a Physics Component (which represent its collision shape),
73 and the engine's corresponding Physics System will update its position according to the physics
74 engine's result.
75 
76 ## Frame and Tasks
77 
78 At each frame, the main loop first starts the renderer, then each System is given the opportunity
79 to add _Tasks_ to the task queue. Tasks are processed in parallel during every frame. They can have
80 dependencies between them (e.g. the physics update can wait for the animation update to complete).
81 
82 A Task must implement the interface `Core::Task` which defines a `process()` function which will be called
83 when the task is executed. However for most cases, it is usually convenient to use a `FunctionTask` which takes
84 an instance of `std::function` (which will be called by the `process()` method). Calling an object's member function
85 in a task is thus easily achieved by using `std::bind()` to create the function object.
86 
87 When a task is added, the task queue assumes ownership of the task and returns a `TaskID`
88 which identifies the task.
89 
90 Dependencies between tasks can be specified either as _immediate_ dependencies or _pending_ dependencies.
91 Immediate dependencies are dependencies between tasks which are already present in the task queue.
92 Adding a dependency with `TaskkQueue::addDependency()` takes two arguments, the predecessor and the successor task,
93 and the task queue will ensure that the successor task must be executed after all its predecessors have finished.
94 In `addDependency()`, tasks can be identified with either a `TaskID` or a name; at least one of
95 the arguments must be a `TaskID`.
96 If a name is given, _all_ tasks matching with the name will be added as dependencies.
97 
98 Since the order in which the systems are called is unspecified, a system may want
99 to add a dependency to a task that is not yet present. Pending dependencies solve
100 this problem by specifying a dependency between a `TaskID`and a name (which may not be present
101 in the task queue).
102 Pending dependencies are resolved just before the task queue starts executing all tasks.

104 ## Rendering
105 
106 Each Component may have some _RenderObjects_ (aka Drawable) which are (usually) OpenGL objects. When a Component has
107 changed it needs to tell its drawable to update their internal data (such as OpenGL VBOs).
108 Render objects are stored together by the _render object manager_ for efficiency, thus components only store an index to reference their render objects.
109 
110 The Render Objects are drawn by the _Renderer_ which may live in a separated thread. Each frame it grabs all
111 the drawable (which should be double-buffered in case we are mid-VBO update) and calls `draw()` on them.

115 ## ItemEntry : Handles to engine objects
116 
117 The structure `ItemEntry` acts as a general identifier for all three levels of engine objects (entities, components and render objects) by storing a handle to the three objects (a pointer or an index for render objects). An entry representing an entity will have its entity handle set and the other two invalid; an entry representing a component will have the component handle set and the entity handle set to the entity owning the component. Similarly a render object entry will have all three handles sets .
118 Any other combination is invalid.
119 
120 Item entries are mainly used to harmonize the interface to manipulate objects in the GUI.
121 
122 ## Object manipulation and editable transforms
123 
124 Components have an interface through which transforms can be edited with three functions :
125 * `canEdit()`
126 * `getTransform()`
127 * `setTransform()`
128 
129 Each of these functions takes a render object index as argument, as sometimes a component exposes multiple objects whose
130 transform can be edited. Function `canEdit()` tells whether a given idnex can be used as a handle for the other two functions.
131 
132 ## The system Entity and Debug Display
133 
134 One Entity is always created when the engine starts, called the System Entity. Implemented as a singleton, this entity acts as a special object that the engine can use internally to display elements not related to any other Entity. The System Entity is fixed to the world and cannot be moved.
135 
136 In particular it is expected that UI elements (e.g. Gizmos) should be attached to the UI component. The Debug Component is useful to display debug primitives which can be useful for graphic debugging.
137 A series of macros `RA_DISPLAY_...` are defined to conveniently add basic display objects such as points, vectors, lines or 3D frames from anywhere in the code. A compile-time switch (`RA_DISABLE_DEBUG_DISPLAY`) can be activated to disable this feature.

139 # The plugin framework
140 
141 Entities and components have been designed so that the engine is modular in terms of features. It is expected that most interesting works will be done by Systems defined in _Plugins_. Each plugin can define its System (and the corresponding Components).
142 
143 We use a compile-time plugins loading mechanism. When running `cmake`, it will list the contents of the `src/Plugins/` directory and add them to be compiled with the project, and automatically generate the code to include the plugins Systems in the main application.
144 
145 ## Requirements
146 
147 For this automated build to work the plugins are required to follow these requirements
148 * The plugin code must be in a sub-folder of the `src/Plugins/` directory.
149 
150 * The folder name is taken to be the _base name_ of the plugin. If the directory name is `BaseName`, then
151  * Plugin code must be in `namespace BaseNamePlugin`
152  * The system defined by the plugin must be named `BaseNamePlugin::BaseNameSystem`
153  * It must be defined in a header located at the top of the plugin sub-folder, named `BaseNameSystem.hpp` (its full path should be `src/Plugins/Basename/BasenameSystem.hpp`)
154  * The system must inherit from `Engine::System` and have a default empty constructor.
155 
156 ## Default plugins

158 See the structure of the default plugins for an example of a working plugin.

160 * _Animation_ handles an animation skeleton and can play keyframe animations.

162 * _Skinning_ uses a geometric skinning method (like linear blend skinning or dual quaternion).
