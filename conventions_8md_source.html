7 # Headers #
8 
9 * Every .cpp must have an associated .hpp file
10 * use .inl files for inline functions definitions and include it in the .hpp
11 * Two types of include guards are accepted (modern is preferred):
12  * Legacy: `#ifndef HEADER_NAME_HPP_`
13  * Modern: `#pragma once`
14 * Every class should have its own header.
15 * Always use < > in include directives and never " "
16 * Keep headers in order : Class header, system libraries, other libraries, other headers from project.
17 * forward declare as much as you can
18 
19 # Functions
20 * use const reference or value to pass input parameters
21 * use references for passing output parameters
22 * output parameters appear last.
23 
24 # Scope and names
25 * use anonymous namespaces for file local variables / functions. no classes with only static functions !
26 * Use `namespace Ra { }` for all radium engine code, plus a sub-namespace for each module (e.g. `Ra::Physics`)
27 * never use the `using` instruction for a whole namespace
28 * put all non member functions in a namespace (no top level global functions)
29 * declare pointers and ref attached to their type : `void* a`, `const Foo& bar`
30 * macros and defined constants should be in capitals.
31 
32 # Variables
33 * declarations should always be made on separate lines; (no `int a, b, c;`)
34 * initialize variables on the line of declaration whenever possible
35 * no global variables. If really necessary, prefix with `g_`
36 * use C++11 `nullptr` for null pointers.
37 * use `auto` only when it helps readability.
38 
39 # Scalar types
40 * Radium defines a default type `Scalar`, set either as `float` or `double` depending on the cmake
41 option `RADIUM_WITH_DOUBLE_PRECISION`
42 * Always use `Scalar` type to represent floating point numbers, except when interfacing with external
43 libraries using a fixed floating point type (e.g. Qt).
44 * Radium offers operators to create `Scalar` from integer and floating point numbers: always
45 use `Scalar()` or `_ra` suffix when defining numbers from literals (e.g. `auto a = .5_ra;`,
46 `Scalar b = a * Scalar( 2 );` or `Scalar c = a / 2_ra;`).
47 * Equality between Scalar values needs to be computed using `Ra::almost_equals` (see CoreMacros.hpp).

49 # Code style
50 * Indentation style : 4-spaces
51 * Brace style : keep it consistent across files.
52 * Case style : CamelCase
53 * Only classes have their first letter capitalized. Functions and variables don't.
54 * Class members have the `m_` prefix. Other prefixes (apart from `g_` for globals, and `s_` for static classe members) are discouraged.
55 * no ifs / for / while one-liners. Braces everywhere. Even for cases in a switch.
56 * separate different clauses in a boolean expression with parens.
57 `( (a || b) && ( c+d < 0) )`
58 * use c++ style casts unless converting a value (e.g. `int x = int(f))`. Avoid `const_cast` if possible.
59 * use prefix increments (`++i` and not `i++`)
60 * use `const` everywhere possible. use `constexpr` for const values.
61 * line length should be kept at 80 (soft limit) and not exceed 120 (hard limit)
62 * no need for () for a return statement.
63 
64 # Class design
65 * Constructors should be trival. All complex work goes in an `init()` function
66 * Try to order class members by size (biggest to smallest)
67 * Any class containing a fixed-size `Ra::Core::Vector` or `Matrix`member must declare `RA_CORE_ALIGNED_NEW`
68 * Try to declare these fixed-size vectors all together, and preferably first in the class.
69 * Never pass a fixed-size vector by value to a function (only ref and const ref). Using them as return value is fine.
70 * use `explicit` for all one-arguments constructors
71 * use `= delete` for preventing copy constructor and other compiler-generated functions
72 * use `= default` for default compiler-generated function (copy constructor, assignment operator)
73 * use `struct` for trivial PODs where all data is public and `class` for
74 everything else.
75 * no other inheritance than public
76 * if multiple inheritance, all parent classes but one have to be interfaces (= pure virtual classes)
77 * composition is better than inheritance
78 * if the class has virtual methods then the destructor is virtual
79 * use `override` when overriding a virtual method.
80 * use operator overloading only if the meaning is non-ambiguous (only for arithmetic-like syntax).
81 * be consistent with operators (i.e. if you overload + and = also overload +=).
82 * declare : public functions (constructors first), private functions, public variables, private variables.
83 * declare nested types and enums before functions
84 * do not abuse function overloading.
85 * avoid `mutable`
86 * getters and setter should have a consistent name with the variable.
87 
88 # Non-negociable
89 * No exceptions, no RTTI, no dynamic cast, no gotos.
